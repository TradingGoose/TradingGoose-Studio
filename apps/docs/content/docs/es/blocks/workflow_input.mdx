El bloque Flujo de trabajo ejecuta otro flujo de trabajo (hijo) desde dentro del flujo de trabajo actual (padre). Lee el esquema del **Disparador de Formulario de Entrada** del flujo hijo y presenta una interfaz de mapeo para que puedas pasar variables del padre a las entradas esperadas del hijo. Este es el mecanismo principal para modularizar automatizaciones complejas en sub-flujos de trabajo reutilizables.

## Descripción general

<Steps>
  <Step>
    <strong>Seleccionar un flujo de trabajo hijo</strong>: Elige cualquier flujo de trabajo en tu espacio de trabajo (excepto el actual)
  </Step>
  <Step>
    <strong>Mapear entradas</strong>: El bloque lee el Disparador de Formulario de Entrada del hijo y muestra cada campo esperado. Mapea variables del padre o define valores fijos para cada uno.
  </Step>
  <Step>
    <strong>Ejecutar</strong>: El flujo de trabajo hijo se ejecuta hasta completarse con las entradas mapeadas
  </Step>
  <Step>
    <strong>Devolver resultados</strong>: La salida final del hijo se devuelve al padre como la salida `result` del bloque
  </Step>
</Steps>

## Cómo funciona

El flujo de trabajo hijo debe utilizar un **Disparador de Formulario de Entrada** como bloque de inicio. Ese disparador define el esquema de entrada (nombres de campo, tipos y descripciones). Cuando seleccionas el flujo de trabajo hijo en este bloque, la interfaz de mapeo de entradas renderiza dinámicamente esos campos para que puedas conectarlos.

En tiempo de ejecución, el bloque:

1. Resuelve todos los valores mapeados del estado actual del flujo de trabajo padre.
2. Invoca el ejecutor del flujo de trabajo hijo, pasando las entradas resueltas.
3. Espera a que el hijo finalice.
4. Devuelve `success`, `childWorkflowName` y el `result` completo de la ejecución del hijo.

<Callout type="info">
  El desplegable solo muestra los flujos de trabajo en tu espacio de trabajo que son diferentes al actual. No es posible invocar un flujo de trabajo de forma recursiva.
</Callout>

### Campos

| Campo | Requerido | Descripción |
|---|---|---|
| **Seleccionar flujo de trabajo** | Sí | El flujo de trabajo hijo a ejecutar. Debe tener un Trigger de formulario de entrada. |
| **Mapeo de entradas** | -- | Mapeo dinámico de valores del padre al esquema del trigger del hijo. Aparece después de seleccionar un flujo de trabajo. |

## Salidas

| Salida | Tipo | Descripción |
|---|---|---|
| `success` | boolean | Indica si el flujo de trabajo hijo se completó correctamente |
| `childWorkflowName` | string | Nombre del flujo de trabajo hijo ejecutado |
| `result` | json | El resultado completo de la ejecución del flujo de trabajo hijo |
| `error` | string | Mensaje de error si el flujo de trabajo hijo falló |

## Mejores prácticas

- **Diseña los flujos de trabajo secundarios con Disparadores de Formulario de Entrada claros** -- nombra los campos de forma descriptiva para que la interfaz de mapeo sea fácil de usar en el flujo principal.
- **Mantén los flujos de trabajo secundarios enfocados** -- un flujo de trabajo secundario debe hacer una sola cosa bien, haciéndolo reutilizable en múltiples flujos principales.
- **Verifica la salida de `success`** -- usa un bloque de Condición después del bloque de Flujo de trabajo para gestionar los fallos de forma adecuada.
