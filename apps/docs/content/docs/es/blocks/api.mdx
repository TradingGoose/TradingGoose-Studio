## Descripción general

El bloque API le permite:

<Steps>
  <Step>
    <strong>Conectarse a servicios externos</strong>: Realizar solicitudes HTTP a APIs REST y servicios web
  </Step>
  <Step>

  </Step>
  <Step>

  </Step>

## Cómo Funciona

El bloque API procesa las solicitudes HTTP mediante un enfoque estructurado:

1. **Configurar Solicitud** - Establece la URL, el método, los encabezados y los parámetros del cuerpo
2. **Ejecutar Solicitud** - Envía la solicitud HTTP al endpoint especificado
3. **Procesar Respuesta** - Gestiona los datos de respuesta, los códigos de estado y los encabezados
4. **Manejo de Errores** - Administra tiempos de espera, reintentos y condiciones de error

## Opciones de Configuración

### URL

La URL del endpoint para la solicitud API. Puede ser:

- Una URL estática introducida directamente en el bloque
- Una URL dinámica conectada desde la salida de otro bloque
- Una URL con parámetros de ruta

### Método

Selecciona el método HTTP para tu solicitud:

- **GET**: Recuperar datos del servidor
- **POST**: Enviar datos al servidor para crear un recurso
- **PUT**: Actualizar un recurso existente en el servidor
- **DELETE**: Eliminar un recurso del servidor
- **PATCH**: Actualizar parcialmente un recurso existente

### Parámetros de Consulta

Define pares clave-valor que se añadirán a la URL como parámetros de consulta. Por ejemplo:

```
Key: apiKey
Value: your_api_key_here

Key: limit
Value: 10
```

Estos se agregarían a la URL como `?apiKey=your_api_key_here&limit=10`.

### Encabezados

Configura los encabezados HTTP para tu solicitud. Los encabezados más comunes incluyen:

```
Key: Content-Type
Value: application/json

Key: Authorization
Value: Bearer your_token_here
```

### Cuerpo de la Solicitud

Para los métodos que admiten un cuerpo de solicitud (POST, PUT, PATCH), puedes definir los datos a enviar. El cuerpo puede ser:

- Datos JSON introducidos directamente en el bloque
- Datos conectados desde la salida de otro bloque
- Generados dinámicamente durante la ejecución del flujo de trabajo

### Acceso a los Resultados

Una vez completada una solicitud API, puedes acceder a sus salidas:

- **`<api.data>`**: Los datos del cuerpo de respuesta de la API
- **`<api.status>`**: Código de estado HTTP (200, 404, 500, etc.)
- **`<api.headers>`**: Encabezados de respuesta del servidor
- **`<api.error>`**: Detalles del error si la solicitud falló

## Funciones Avanzadas

### Construcción Dinámica de URLs

Construya URLs dinámicamente usando variables de bloques anteriores:

```javascript
// In a Function block before the API
const userId = <start.userId>;
const apiUrl = `https://api.example.com/users/${userId}/profile`;
```

### Reintentos de Solicitud

El bloque de API maneja automáticamente:
- Tiempos de espera de red con retroceso exponencial
- Respuestas de límite de velocidad (códigos de estado 429)
- Errores del servidor (códigos de estado 5xx) con lógica de reintento
- Fallos de conexión con intentos de reconexión

### Validación de Respuesta

Valide las respuestas de la API antes de procesarlas:

```javascript
// In a Function block after the API
if (<api.status> === 200) {
  const data = <api.data>;
  // Process successful response
} else {
  // Handle error response
  console.error(`API Error: ${<api.status>}`);
}
```

## Mejores Prácticas

- **Usa variables de entorno para datos sensibles**: No codifiques de forma fija las claves de API ni las credenciales
- **Gestiona los errores de forma adecuada**: Conecta la lógica de manejo de errores para las solicitudes fallidas
- **Valida las respuestas**: Comprueba los códigos de estado y los formatos de respuesta antes de procesar los datos
- **Respeta los límites de frecuencia**: Ten en cuenta los límites de frecuencia de la API e implementa una limitación de solicitudes adecuada
