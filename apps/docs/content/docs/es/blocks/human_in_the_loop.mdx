El bloque Human in the Loop pausa un flujo de trabajo en ejecución y espera a que una persona revise, proporcione información o apruebe antes de continuar la ejecución. Es el mecanismo principal para añadir supervisión humana a los pipelines automatizados.

## Descripción general

<Steps>
  <Step>
    <strong>Pausar la ejecución</strong>: El flujo de trabajo se detiene en este bloque y entra en estado de espera
  </Step>
  <Step>
    <strong>Presentar contexto</strong>: Se muestran datos estructurados al revisor para que pueda tomar una decisión informada
  </Step>
  <Step>
    <strong>Notificar a los aprobadores</strong>: Opcionalmente, se envía un mensaje de Slack, un correo electrónico u otra notificación con la URL de revisión
  </Step>
  <Step>
    <strong>Recopilar información y reanudar</strong>: El aprobador completa el formulario de reanudación y el flujo de trabajo continúa con su información disponible en etapas posteriores
  </Step>
</Steps>

## Cómo funciona

Cuando el ejecutor alcanza un bloque Human in the Loop:

1. **Renderiza los datos de visualización** -- Los campos `builderData` se evalúan (incluidas las referencias a `<variable.name>`) y se presentan como un resumen estructurado para el revisor.
2. **Genera una URL de reanudación** -- Se crea una URL única y un endpoint de API. La URL abre una interfaz de revisión; el endpoint acepta un `POST` programático para reanudar.
3. **Envía notificaciones** -- Si se ha configurado una herramienta de notificación (Slack, correo electrónico, etc.), la URL de reanudación se envía automáticamente.
4. **Espera** -- La ejecución queda suspendida. Ningún bloque posterior se ejecuta hasta que el revisor actúe.
5. **Reanuda** -- Una vez que el revisor envía el formulario de reanudación, el flujo de trabajo continúa desde donde se detuvo. Todos los campos completados por el revisor están disponibles como salidas del bloque.

<Callout type="info">
  El bloque genera dos valores tras pausarse: **url** (el enlace a la interfaz de revisión) y **resumeEndpoint** (una URL de API a la que puedes llamar con `curl` o cualquier cliente HTTP para reanudar el flujo de forma programática).
</Callout>

### Campos

| Campo | Propósito |
|---|---|
| **Mostrar datos** | Datos estructurados en clave-valor mostrados al revisor. Referencia variables previas del flujo con la sintaxis `<block.output>`. |
| **Notificación** | Adjunta una herramienta (Slack, Email, etc.) que se activará cuando el bloque se pause, enviando la URL de reanudación a las personas correspondientes. |
| **Formulario de reanudación** | Define los campos de entrada que el aprobador debe completar antes de que el flujo de trabajo se reanude. Estos se convierten en salidas del bloque. |

## Salidas

| Salida | Tipo | Descripción |
|---|---|---|
| `url` | string | URL a la interfaz de revisión donde el aprobador puede actuar |
| `resumeEndpoint` | string | Endpoint de API para reanudar el flujo de trabajo mediante HTTP POST |

## Mejores prácticas

- **Siempre configure una notificación** para que los aprobadores sepan que hay un flujo de trabajo esperando su atención. Sin ella, la URL de reanudación debe compartirse manualmente.
- **Mantenga los datos de visualización enfocados** -- incluya solo la información que el revisor necesita para tomar una decisión.
- **Diseñe el formulario de reanudación con cuidado** -- los bloques posteriores dependerán de los campos que defina aquí, así que nómbrelos con claridad.
