## Disparadores principales

Selecciona un disparador por flujo de trabajo para definir cómo se inicia:

<Cards>
  <Card title="API" href="/triggers/api">
    Endpoint HTTP que mapea cuerpos JSON como entradas del flujo de trabajo
  </Card>
  <Card title="Chat" href="/triggers/chat">
    Interfaz de chat desplegada con respuestas en streaming
  </Card>
  <Card title="Input Form" href="/triggers/input-form">
    Entrada manual tipada utilizada en ejecuciones del editor y flujos de trabajo secundarios
  </Card>
  <Card title="Manual" href="/triggers/manual">
    Ejecuciones bajo demanda sin datos adicionales
  </Card>
  <Card title="Schedule" href="/triggers/schedule">
    Ejecución basada en cron o intervalos
  </Card>
  <Card title="Webhook" href="/triggers/webhook">
    Recibe payloads de webhooks externos
  </Card>
</Cards>

## Comparación rápida

| Disparador | Condición de inicio |
|---------|-----------------|
| **API** | HTTP POST autenticado |
| **Chat** | Mensaje en el despliegue de chat |
| **Input Form** | Al enviar manualmente en el editor o flujo de trabajo padre |
| **Manual** | Botón de ejecución en el editor |
| **Schedule** | Temporizador gestionado en el modal de programación |
| **Webhook** | Al recibir una solicitud HTTP entrante |

## Uso de disparadores

1. Coloca el bloque disparador en la ranura de inicio.
2. Configura el esquema o la autenticación requeridos.
3. Conecta el bloque al resto del flujo de trabajo.

> Los despliegues potencian cada disparador. Actualiza el flujo de trabajo, vuelve a desplegar y todos los puntos de entrada del disparador adoptarán el nuevo snapshot. Más información en [Ejecución → Snapshots de despliegue](/execution).

Los bloques Starter heredados permanecen disponibles para flujos existentes, pero ya no aparecen en nuevas compilaciones.
