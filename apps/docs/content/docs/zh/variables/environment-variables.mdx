环境变量提供了一种安全的方式，用于管理工作流中的配置值和密钥，包括 API 密钥及工作流运行时所需的其他敏感数据。它们将密钥从工作流定义中剥离，同时确保在执行期间可以访问这些值。

## 变量类型

TradingGoose 中的环境变量在两个层级上生效：

- **个人环境变量**：仅对您的账户私有，只有您可以查看和使用
- **工作区环境变量**：在整个工作区内共享，所有团队成员均可使用

<Callout type="info">
当存在命名冲突时，工作区环境变量优先于个人环境变量。
</Callout>

## 配置环境变量

前往 **设置** 以配置您的环境变量。在工作区设置中，您可以创建和管理个人及工作区级别的环境变量。个人变量仅对您的账户私有，而工作区变量则与所有团队成员共享。

### 设置变量为工作区范围

使用工作区范围切换开关，可将变量共享给整个团队。启用工作区范围后，该变量将对所有工作区成员可见，并可在该工作区内的任意工作流中使用。

### 工作区变量视图

配置工作区范围变量后，这些变量将与您的个人变量一同显示在环境变量列表中，并清晰标注其所属范围。

## 在工作流中使用变量

要在工作流中引用环境变量，请使用 `{{}}` 表示法。在任意输入字段中输入 `{{` 时，将弹出下拉菜单，显示您的个人及工作区级别环境变量。直接选择所需变量即可。

## 变量优先级

当您同时拥有同名的个人变量和工作区变量时：

1. **工作区变量优先于**个人变量
2. 这可以防止命名冲突，并确保团队工作流程中行为的一致性
3. 如果存在同名工作区变量，则同名个人变量将被忽略

<Callout type="warning">
请谨慎选择变量名称，以避免意外覆盖。建议为个人变量添加您的姓名首字母前缀，或为工作区变量添加项目名称前缀。
</Callout>

## 安全最佳实践

### 敏感数据处理
- 将 API 密钥、令牌和密码存储为环境变量，而非硬编码
- 对多名团队成员共享的资源使用工作区变量
- 将个人凭据保存在个人变量中

### 变量命名规范
- 使用描述性名称：`DATABASE_URL`，而非 `DB`
- 在团队内遵循统一的命名规范
- 考虑使用前缀以避免冲突：`PROD_API_KEY`、`DEV_API_KEY`

### 访问控制
- 工作区环境变量遵循工作区权限设置
- 只有具有写入权限或更高权限的用户才能创建或修改工作区变量
- 个人变量始终对个人用户保持私密
