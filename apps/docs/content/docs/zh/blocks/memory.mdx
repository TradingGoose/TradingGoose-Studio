记忆块（Memory Block）用于存储和检索在工作流执行之间持久保存的对话消息。它专为与智能体块（Agent Block）协同工作而设计，为您的智能体提供跨运行周期的长期对话上下文。

## 概述

<Steps>
  <Step>
    <strong>添加记忆</strong>：以指定角色（用户、助手或系统）将消息存储至命名的记忆 ID 下
  </Step>
  <Step>
    <strong>获取记忆</strong>：通过 ID 检索单条记忆条目
  </Step>
  <Step>
    <strong>获取所有记忆</strong>：获取当前工作流中存储的全部记忆
  </Step>
  <Step>
    <strong>删除记忆</strong>：通过 ID 删除指定记忆条目
  </Step>
</Steps>

## 工作原理

记忆条目以聊天消息的形式存储，包含三个必填字段：用于划定记忆范围的 **ID**、**角色**（`user`、`assistant` 或 `system`），以及 **内容** 文本。当您稍后检索记忆并将其传入 Agent 模块的记忆输入时，Agent 会将其视为先前的对话轮次，从而在不同执行之间实现多轮对话。

典型的对话工作流在 Agent 前后使用三个 Memory 模块：

1. **Memory（添加，角色：user）** -- 在 Agent 运行前保存传入的用户消息
2. **Agent** -- 以检索到的记忆作为上下文处理消息
3. **Memory（添加，角色：assistant）** -- 在 Agent 运行后保存其响应

<Callout type="info">
  记忆 ID 用于划定对话范围。在多次执行中使用相同的 ID 可构建连续的对话历史，使用不同的 ID 则可维护独立的对话线程。
</Callout>

## 输出

| 输出 | 类型 | 描述 |
|---|---|---|
| `memories` | json | 检索到的记忆数据（消息数组） |
| `id` | string | 被操作的记忆标识符 |

## 最佳实践

- **使用一致的 ID** -- 记忆 ID 是将消息归组为一次对话的关键。使用稳定的标识符（例如用户 ID 或会话 ID），以确保记忆在多次运行中正确累积。
- **存储对话双方的内容** -- 在 Agent 之前添加 `user` 角色消息，在其之后添加 `assistant` 角色消息，以保留完整的对话记录。
- **按需清理** -- 使用删除操作清除过期对话，防止记忆无限增长。
