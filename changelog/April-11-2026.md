## feat/admin-panel vs origin/staging

### Comparison Context

- Head branch: `feat/admin-panel`
- Base branch: `origin/staging`
- Merge base: `e8f67b8649e10c43827bc9f75193f5af7e07c8ca`
- Compare range: `e8f67b8649e10c43827bc9f75193f5af7e07c8ca..feat/admin-panel`
- Working tree note: the branch was dirty during review, so the notes below cover the committed `merge-base..HEAD` history plus the current unstaged billing/auth/execution edits that were part of the same feature pass.

### What Changed

- Public registration is now controlled by platform settings instead of assuming open signup. The durable data model lives in `packages/db/schema/system.ts` via `registrationModeEnum`, `systemSettings.registrationMode`, and the `waitlist` table. The server-side gatekeeping lives in `apps/tradinggoose/lib/registration/service.ts`, while `apps/tradinggoose/lib/registration/shared.ts` owns the `RegistrationMode`, `WaitlistStatus`, button labels, and signup or waitlist href selection. Future auth or landing work should reuse those shared helpers instead of hardcoding signup paths or status strings.
- Signup, waitlist, and admin review all point at that same registration contract. `apps/tradinggoose/app/api/registration/route.ts` exposes the current mode for rendering and client queries, `apps/tradinggoose/app/api/waitlist/route.ts` accepts self-service requests, and `apps/tradinggoose/app/api/admin/registration/route.ts` handles mode changes plus bulk waitlist approval or rejection. The user-facing surfaces are `apps/tradinggoose/app/(auth)/signup/page.tsx`, `apps/tradinggoose/app/(auth)/signup/signup-form.tsx`, `apps/tradinggoose/app/(auth)/waitlist/page.tsx`, and `apps/tradinggoose/app/(auth)/waitlist/waitlist-form.tsx`, with the admin queue in `apps/tradinggoose/app/admin/registration/registration-admin.tsx`.
- Waitlist mail delivery is part of the same flow and is not a separate ad hoc feature. `apps/tradinggoose/lib/registration/service.ts` sends waitlist confirmation and approval emails through `apps/tradinggoose/components/emails/render-email.ts`, which now renders `WaitlistConfirmationEmail` and `WaitlistApprovedEmail`. Future waitlist or onboarding work should add templates through `render-email.ts` and `components/emails/index.ts` instead of sending custom email bodies inline.
- System administration is now a first-class area, with first-admin bootstrap built into the auth flow. `apps/tradinggoose/lib/admin/access.ts` owns `claimFirstSystemAdmin`, `getSystemAdminAccess`, and the advisory-lock bootstrap behavior. `apps/tradinggoose/app/admin/layout.tsx` and `apps/tradinggoose/app/admin/page.tsx` treat "existing system admin" and "first user allowed to bootstrap the admin role" as the only supported entry conditions. Future admin pages should plug into `getSystemAdminAccess()` instead of rolling their own authorization gates.
- Billing moved to DB-backed system tiers and settings instead of scattered constants. The durable schema now lives in `packages/db/schema/system.ts` via `systemBillingSettings` and `systemBillingTier`, while workspace billing ownership lives in `packages/db/schema/core.ts` via `workspace.billingOwnerType`, `workspace.billingOwnerUserId`, `workspace.billingOwnerOrganizationId`, and `workspace.allowPersonalApiKeys`. This branch also adds the admin tier editor and settings surface in `apps/tradinggoose/app/admin/billing/billing-admin.tsx` plus `apps/tradinggoose/app/api/admin/billing/*`.
- Workspace billing ownership now has a single canonical helper layer. `apps/tradinggoose/lib/workspaces/billing-owner.ts` owns `workspaceBillingOwnerSchema`, `toWorkspaceApiRecord`, `resolveWorkspaceBillingOwnerUpdate`, and the guard rails that keep a billing owner attached to valid admin access. Those helpers are already wired into `apps/tradinggoose/app/api/workspaces/[id]/route.ts`, `apps/tradinggoose/app/api/workspaces/[id]/permissions/route.ts`, and `apps/tradinggoose/app/api/organizations/[id]/workspaces/route.ts`. Future branches should reuse those helpers instead of re-serializing `billingOwnerType` fields or duplicating removal and downgrade checks.
- Runtime billing now has an explicit enablement switch instead of the old default-tier fallback. `apps/tradinggoose/lib/billing/settings.ts` now exposes `isBillingEnabledForRuntime()` from the raw system setting, `apps/tradinggoose/lib/billing/core/subscription.ts` owns `ensureDefaultUserSubscription()` and `backfillDefaultUserSubscriptions()`, and `apps/tradinggoose/lib/auth.ts` plus `apps/tradinggoose/app/api/admin/system-settings/route.ts` use those helpers to provision or backfill user subscriptions when billing turns on. `apps/tradinggoose/app/api/admin/billing/tiers/route.ts`, `apps/tradinggoose/app/api/admin/billing/tiers/[id]/route.ts`, and `apps/tradinggoose/lib/admin/billing/tier-mutations.ts` now keep the default tier active while billing is enabled instead of relying on the deleted `disableBillingIfConfigurationInvalid()` backstop.
- Workspace execution billing now has its own canonical helper layer. `apps/tradinggoose/lib/billing/workspace-billing.ts` owns `resolveWorkspaceBillingContext()`, `toRateLimitBillingScope()`, and `getBillingContextResolutionMessage()`. `apps/tradinggoose/app/api/workflows/[id]/execute/route.ts`, `apps/tradinggoose/background/schedule-execution.ts`, `apps/tradinggoose/background/webhook-execution.ts`, `apps/tradinggoose/lib/logs/execution/logger.ts`, and `apps/tradinggoose/lib/webhooks/processor.ts` now use those helpers so missing subscriptions surface as explicit billing errors instead of silently inventing a default-tier context. Runtime rate limiting and usage checks in `apps/tradinggoose/services/queue/RateLimiter.ts`, `apps/tradinggoose/lib/execution/concurrency-limit.ts`, `apps/tradinggoose/app/api/v1/middleware.ts`, `apps/tradinggoose/lib/billing/core/billing.ts`, `apps/tradinggoose/lib/billing/core/usage.ts`, `apps/tradinggoose/lib/billing/calculations/usage-monitor.ts`, and `apps/tradinggoose/lib/billing/storage/limits.ts` now short-circuit or return permissive limits when billing is off, and `apps/tradinggoose/lib/billing/subscriptions/utils.ts` adds `BILLING_ENTITLED_SUBSCRIPTION_STATUSES` so `past_due` still counts as entitled for those checks.
- Public pricing, hosted copy, and SEO now flow from the same public billing catalog. `apps/tradinggoose/lib/billing/catalog.ts` builds the server catalog, `apps/tradinggoose/lib/billing/public-catalog.ts` defines the public DTO and copy helpers, `apps/tradinggoose/app/api/billing/public-catalog/route.ts` exposes it, and `apps/tradinggoose/hooks/queries/public-billing-catalog.ts` is the client query surface. The landing page pricing UI in `apps/tradinggoose/app/(landing)/components/landing-pricing/landing-pricing.tsx`, structured data in `apps/tradinggoose/app/(landing)/components/structured-data.tsx`, and the branded metadata in `apps/tradinggoose/lib/branding/metadata.ts` now all depend on that shared catalog. Future pricing copy should be derived from this catalog instead of being duplicated in page components.
- Subscription upgrade targeting is now explicit about whether a tier belongs to a user or an organization. `apps/tradinggoose/lib/subscription/upgrade-target.ts` resolves the correct `referenceId` and organization activation behavior, while `apps/tradinggoose/lib/subscription/upgrade.ts` is the client entrypoint that talks to Better Auth subscription APIs. Future organization-tier upgrade work should extend those modules rather than reviving the old subscription-transfer flow.
- System-managed OAuth credentials now come from a seeded integration catalog rather than the removed workflow-side credential auto-selector. `apps/tradinggoose/lib/system-integrations/catalog.ts` seeds bundle and service definitions from `OAUTH_PROVIDERS`, `apps/tradinggoose/lib/admin/system-integrations.ts` validates and persists admin edits against that catalog, `apps/tradinggoose/lib/system-integrations/resolver.ts` resolves enabled services plus decrypted secrets, and `apps/tradinggoose/lib/oauth/system-managed-config.ts` hydrates runtime OAuth client credentials from the resolved system bundle or environment fallback. The Better Auth entrypoint `apps/tradinggoose/app/api/auth/[...all]/route.ts` is now the supported place where those credentials are loaded before callback or link handling.
- Organization-scoped SSO registration is now issuer-discovery only. `apps/tradinggoose/app/api/auth/sso/register/route.ts` rejects legacy OIDC endpoint override fields and fills `authorizationEndpoint`, `tokenEndpoint`, `userInfoEndpoint`, and `jwksEndpoint` from the issuer discovery document. Future SSO work should use that route contract and not revive manual endpoint overrides or the removed provider-registration scripts.
- Trigger metadata is aligned to the newer service-specific OAuth naming and webhook-provider contracts. `apps/tradinggoose/triggers/types.ts` documents that trigger and config-field `oauthProvider` values should use service or provider ids, and `apps/tradinggoose/triggers/index.ts` keys native-trigger behavior off `webhookProvider`. A concrete example is `apps/tradinggoose/triggers/gmail/poller.ts`, which now uses the service id `google-email` for its credential subblock. Future trigger additions should follow that service-id pattern instead of inventing another provider lookup layer.
- Provider catalog lookup for AI is now split cleanly between UI metadata and runtime execution. `apps/tradinggoose/providers/ai/utils.ts` builds the browser-safe provider catalog from `providers/ai/models`, while `apps/tradinggoose/providers/ai/index.ts` remains the server execution path. Future UI or settings work should use the utilities layer for model/provider metadata instead of importing runtime providers directly.
- Workspace navigation is now layout-owned instead of being recreated per page. `apps/tradinggoose/app/workspace/layout.tsx` wraps workspace pages with `GlobalNavbar`, and `apps/tradinggoose/global-navbar/use-workspace-switcher.ts` resets the workflow registry when a user switches workspaces. Future workspace pages should rely on the layout wrapper and switcher hook instead of adding another navbar or duplicating workspace-switch side effects.
- Robinhood brokerage support was intentionally removed from the supported trading provider set. The deleted surface is `apps/tradinggoose/providers/trading/robinhood/*`, and the live provider registry now sits in `apps/tradinggoose/providers/trading/index.ts` and `apps/tradinggoose/providers/trading/providers.ts` with Alpaca and Tradier definitions only. Any future trading feature should target the remaining provider contracts instead of reviving Robinhood-specific request builders or normalization code.
- Release automation, image publishing, and SEO metadata were tightened alongside the product changes. `.github/workflows/release.yml` now builds release notes from commits in merged PRs targeting `staging`, deduplicates by commit hash, categorizes notes, adds contributors, and still releases against `staging`. `.github/workflows/images.yml` adds image publishing for app, db migrations, and realtime services with AMD64 ECR pushes and GHCR multi-arch manifests. Landing and blog SEO updates route through `apps/tradinggoose/lib/branding/metadata.ts` and `apps/tradinggoose/app/(landing)/components/structured-data.tsx` rather than bespoke page-level copy.

### Reuse These Contracts

- Reuse `apps/tradinggoose/lib/registration/shared.ts` for registration-mode literals, CTA labels, and auth-route decisions. Do not duplicate mode enums or signup-routing logic in auth pages or marketing components.
- Reuse `apps/tradinggoose/lib/registration/service.ts` for eligibility checks, waitlist mutation rules, invitation bypass, and email dispatch. Public and admin routes should stay thin wrappers over this service.
- Reuse `apps/tradinggoose/lib/workspaces/billing-owner.ts` whenever a workspace API, permission change, or organization workflow touches billing ownership. The API shape should stay on the serialized `billingOwner` object returned by `toWorkspaceApiRecord()`.
- Reuse `apps/tradinggoose/lib/billing/settings.ts` for runtime billing enablement. `getResolvedBillingSettings()` and `isBillingEnabledForRuntime()` are the canonical checks for whether code should meter, rate-limit, or skip billing work.
- Reuse `apps/tradinggoose/lib/billing/core/subscription.ts` for default-user provisioning. `ensureDefaultUserSubscription()` and `backfillDefaultUserSubscriptions()` own the bootstrap path when billing turns on.
- Reuse `apps/tradinggoose/lib/billing/workspace-billing.ts` for execution billing context, rate-limit scope serialization, and billing error messages. `resolveWorkspaceBillingContext()`, `toRateLimitBillingScope()`, and `getBillingContextResolutionMessage()` keep workflow, webhook, and rate-limit paths aligned.
- Reuse `apps/tradinggoose/lib/billing/subscriptions/utils.ts` for entitlement checks. `BILLING_ENTITLED_SUBSCRIPTION_STATUSES` is the canonical set when a path should still treat `past_due` as live.
- Reuse `apps/tradinggoose/app/api/auth/sso/register/route.ts` for issuer-discovery-only org SSO registration. Keep the schema limited to the issuer-based payload and let discovery populate the endpoint fields.
- Reuse `apps/tradinggoose/lib/billing/catalog.ts`, `apps/tradinggoose/lib/billing/public-catalog.ts`, and `apps/tradinggoose/hooks/queries/public-billing-catalog.ts` for any hosted pricing UI, structured data, or docs content. This branch made the public catalog the canonical source for self-serve pricing copy.
- Reuse `apps/tradinggoose/lib/subscription/upgrade-target.ts` and `apps/tradinggoose/lib/subscription/upgrade.ts` for upgrade flows that can land on a user or organization billing account.
- Reuse `apps/tradinggoose/lib/system-integrations/catalog.ts`, `apps/tradinggoose/lib/system-integrations/resolver.ts`, and `apps/tradinggoose/lib/oauth/system-managed-config.ts` for system-managed OAuth work. Provider bundles and secrets now belong to the system integration catalog, not to workflow-local credential inference.
- Reuse `apps/tradinggoose/lib/admin/access.ts` for any new admin area authorization. The bootstrap-admin path is already encoded there and should not be reimplemented in page components.
- Reuse `apps/tradinggoose/providers/ai/utils.ts` for provider catalogs, model lists, and pricing/capability lookups in client-safe code. Keep runtime request execution in `apps/tradinggoose/providers/ai/index.ts`.
- Reuse `apps/tradinggoose/global-navbar/use-workspace-switcher.ts` for workspace switching behavior, especially the workflow-registry reset path.
- Reuse `apps/tradinggoose/lib/billing/tier-summary.ts` for empty/default billing-tier access flags such as `EMPTY_BILLING_TIER_SUMMARY`, `canTierEditUsageLimit`, and `canTierConfigureSso`.
- Reuse `apps/tradinggoose/lib/admin/billing/tier-mutations.ts` and the tier route guards for admin tier validation. Keep the default-tier-active checks in the route handlers instead of duplicating them in page components.

### Do Not Reintroduce

- Do not recreate `apps/tradinggoose/lib/workflows/credential-resolver.ts`. Workflow-local credential auto-selection was replaced by the system integration catalog and `lib/oauth/system-managed-config.ts`.
- Do not recreate `apps/tradinggoose/hooks/use-subscription-state.ts`. Subscription UI should compose Better Auth's `useSubscription()`, the org and billing query hooks, `usePublicBillingCatalog()`, and the pure helpers in `apps/tradinggoose/lib/subscription/helpers.ts`.
- Do not recreate `apps/tradinggoose/lib/subscription/index.ts` as a barrel. Import `helpers.ts`, `types.ts`, `upgrade-target.ts`, and `upgrade.ts` directly so the subscription surface stays explicit.
- Do not recreate `apps/tradinggoose/hooks/queries/sso.ts`. The org-scoped SSO registration route now owns that client flow.
- Do not restore `apps/tradinggoose/lib/billing/settings.ts`'s deleted `disableBillingIfConfigurationInvalid()` helper or any other automatic billing-disable backstop. Use `isBillingEnabledForRuntime()` and the admin system-settings toggle instead.
- Do not reintroduce manual OIDC endpoint override fields into `apps/tradinggoose/app/api/auth/sso/register/route.ts`; issuer discovery is the only supported path for SSO registration.
- Do not hardcode `active` for billing entitlement checks. Use `BILLING_ENTITLED_SUBSCRIPTION_STATUSES`.
- Do not recreate `packages/db/scripts/register-sso-provider.ts` or `packages/db/scripts/deregister-sso-provider.ts`. The org-scoped SSO flow now lives in the app route and Better Auth integration instead of those scripts.
- Do not restore `apps/tradinggoose/lib/billing/constants.ts`. Free-tier defaults, tier access flags, pricing metadata, and billing thresholds now come from `lib/billing/tier-summary.ts`, `lib/billing/tiers.ts`, `lib/billing/public-catalog.ts`, and DB-backed settings in `lib/billing/settings.ts`.
- Do not restore `apps/tradinggoose/providers/trading/robinhood/*` or add Robinhood back into `providers/trading/index.ts` without a deliberate product decision. The supported trading-provider contract for this branch is Alpaca plus Tradier only.
