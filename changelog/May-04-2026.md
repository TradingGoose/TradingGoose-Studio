### Design Decisions
- Monitor ownership moved out of Logs into `/workspace/[workspaceId]/monitor` so execution review and monitor configuration can share saved views, mode switching, and workspace-level nav without overloading `logs` URL state.
- Saved views are strict and mode-aware. `assertStoredMonitorSavedViewConfig()` intentionally rejects mode-less `monitor_view.config` data instead of adding a legacy backfill path, matching the project's no-legacy rule.
- The view model is split by mode because execution views need log grouping, timeline, quick filters, and column limits, while config views need monitor-record dimensions, status lanes, and edit/drop context. Both modes still share `MonitorSavedViewConfig` and saved-view persistence.
- Execution data uses full log detail only where monitor behavior needs trigger monitor snapshots. The regular logs list can keep basic detail, while `useMonitorWorkspaceLogs()` and `useMonitorExecutionSummaries()` request `details: 'full'`.
- Kanban drag/drop is layered. `board/kanban.tsx` owns DnD mechanics and accessibility, `board/monitor-kanban.tsx` owns monitor visual primitives, and execution/config boards own domain-specific reorder and move behavior.
- Config board drag/drop edits monitor records through validated draft patches rather than directly mutating persisted configs. `resolveConfigBoardContextPatch()`, `mergeMonitorDraftPatch()`, and `validateMonitorDraft()` keep provider, interval, listing, workflow target, and status changes consistent with the editor.
- Indicator monitor updates preserve previous auth unless the provider changes, and indicator input overrides are stored only when they differ from metadata defaults through `normalizeIndicatorInputOverrides()`.
- Existing `LayoutTabs` remains the shared tab control, but monitor saved views use an external rename dialog through `onRequestRename` so tab editing does not need a separate inline persistence path.

### Future Branch Guardrails
- Do not add new monitor UI under `apps/tradinggoose/app/workspace/[workspaceId]/logs/logs.tsx` or restore the logs URL `view=monitors`. Extend `apps/tradinggoose/app/workspace/[workspaceId]/monitor/*` instead.

- Do not add legacy fallback for mode-less `monitor_view` rows. If existing data is stale during development, delete or reset the row instead of teaching the app to infer missing mode.
- Keep execution and config mode fields aligned with their allowed constant arrays in `view-config.ts`. UI controls, parser validation, saved-view API tests, and board logic depend on those arrays being canonical.
- Do not fetch monitor execution logs without `details: 'full'` when trigger monitor snapshots are needed. `useMonitorWorkspaceLogs()` and `useMonitorExecutionSummaries()` are the canonical hooks for those reads.
- Do not duplicate kanban drag/drop mechanics in config or execution code. Reuse `board/kanban.tsx` and monitor wrappers from `board/monitor-kanban.tsx`.
- Do not mutate monitor records from board drops without `resolveConfigBoardContextPatch()`, draft validation, and the existing `MonitorRecordActions` path.
- Do not store default indicator input values in monitor configs. Route saved overrides through `normalizeIndicatorInputOverrides()` with metadata from `loadIndicatorInputMetadata()`.
- Do not clear encrypted provider secrets during partial monitor updates. Provider changes may reset auth; same-provider updates should preserve existing encrypted secrets unless replacement values are supplied.
- Keep `/monitor` in workspace navigation and workspace-switch paths through `global-navbar/utils.ts` when changing workspace nav sections.
