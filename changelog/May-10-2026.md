### Design Decisions
- Trading account identity is now the tuple `providerId + credentialServiceId + accountId`, not a provider-specific account label. `apps/tradinggoose/providers/trading/portfolio-identity.ts` owns the canonical equality/key helpers so selectors, sockets, and tool routes cannot drift on comparison logic.
- Provider-route preflight is centralized in `apps/tradinggoose/app/api/providers/trading/shared.ts`. Routes resolve auth once, then validate the chosen account against `listPortfolioIdentities()` before issuing broker requests, which prevents account-detail and holdings flows from hand-rolling provider/account checks.
- Copilot tool approval is metadata-driven. `apps/tradinggoose/lib/copilot/access-policy.ts` and `apps/tradinggoose/stores/copilot/tool-registry.ts` decide whether a tool is gated, while `apps/tradinggoose/lib/copilot/tools/server/router.ts` owns server-tool execution and `apps/tradinggoose/lib/copilot/tools/client/server-tool-metadata.ts` owns the UI copy/icons for those server-backed tools.
- Workflow-editor state stays Yjs-first. `apps/tradinggoose/lib/yjs/use-workflow-doc.ts` recalculates runtime sub-block/output state after writes, and the editor/preview surfaces all reuse `buildSubBlockRows()`, `SubBlockEditRows`, and `SubBlockSummaryRows` instead of maintaining separate preview registries or read-only block renderers.
- Validation is intentionally split. `apps/tradinggoose/serializer/index.ts` only blocks missing `user-only` required inputs before execution begins, while `apps/tradinggoose/tools/params.ts` keeps `user-only` inputs out of LLM schemas so merge-time validation can remain focused on `user-or-llm` and `llm-only` parameters.
- Trigger persistence is explicit. `apps/tradinggoose/triggers/resolution.ts` and `apps/tradinggoose/hooks/use-webhook-management.ts` treat `selectedTriggerId` as the source of truth for trigger-specific fields, webhook saves, and singleton trigger auto-selection.

### Future Branch Guardrails
- Do not pass trading account state around as loose `credentialServiceId` and `accountId` pairs when a `PortfolioIdentity` object is available. Normalize through `toPortfolioValueObject()` and compare with `arePortfolioIdentitiesEqual()`.
- Do not add holdings or broker-detail routes that skip `resolveTradingProviderContext()` or `resolveTradingProviderSelectedAccount()`. Trading account access must stay tied to the selected OAuth service and discovered account list.
- Do not reintroduce client-side wrappers for server-executed Copilot tools or restore `/api/copilot/confirm`. New Copilot tools should be registered in `apps/tradinggoose/stores/copilot/tool-registry.ts`, given display metadata in `apps/tradinggoose/lib/copilot/tools/client/server-tool-metadata.ts`, and routed through `apps/tradinggoose/lib/copilot/tools/server/router.ts` when they execute on the server.
- Do not invent parallel sub-block ids for the same execution parameter. Use `canonicalParamId` in block configs and keep serializer, preview, and execution code reading params through `buildConfiguredSubBlockParams()`.
- Do not mark secret or user-picked values as `user-or-llm` just to avoid early validation. Keep credentials, broker selections, and other user-owned fields `user-only` so serializer and LLM schema generation preserve the current two-stage validation contract.
- Do not add another preview registry or bespoke read-only block renderer. Extend `SubBlockEditRows`, `SubBlockSummaryRows`, `buildSubBlockRows()`, or `ToolSubBlockRenderer` instead.
- Do not bypass `selectedTriggerId` or add fallback trigger resolution based on block type, hidden defaults, or webhook provider config. Extend `apps/tradinggoose/triggers/resolution.ts` and `apps/tradinggoose/hooks/use-trigger-config-aggregation.ts` if trigger ownership rules need to change.
