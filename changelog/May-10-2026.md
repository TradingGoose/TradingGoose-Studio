### Design Decisions
- Trading account identity is the tuple `providerId + credentialServiceId + accountId`. `apps/tradinggoose/providers/trading/portfolio-identity.ts` owns equality and serialization so provider routes, widgets, sockets, and tools do not compare raw labels or account ids independently.
- Order writes belong to the canonical provider order route, not to a generic order-history POST endpoint. `recordOrderHistory()` is reusable persistence infrastructure, while `/api/providers/trading/order` owns broker validation, submission, and recording.
- Workflow execution is queue-first. API, published chat, builder, and child workflow runs all converge on `pendingExecution`, event payloads, `/queue`, `/executions/[executionId]/stream`, and `/api/jobs/[jobId]` so cancellation, event streaming, billing limits, and local Trigger.dev fallback stay consistent.
- Server-to-server workflow calls carry short-lived internal JWT metadata. `InternalWorkflowExecutionContext` identifies workflow-block child executions without restoring legacy browser-only workflow executor tools or broad credential access.
- Copilot tool names now describe read/write behavior. `read_workflow`, `read_workflow_logs`, `read_credentials`, `read_environment_variables`, `get_available_blocks`, and `get_agent_accessory_catalog` are the stable contracts; registry metadata decides whether execution is client-managed, server-managed, or gated.
- Workflow editor state remains Yjs-first and sub-block-driven. Serialization, deploy previews, read-only panels, block output selectors, and trigger hydration all reuse shared row/value/output helpers instead of separate preview registries or block-specific fallback paths.

### Future Branch Guardrails
- Do not pass trading account state as loose labels, `credentialServiceId`/`accountId` pairs, or provider-specific account records when a `PortfolioIdentity` object is available. Normalize with `toPortfolioValueObject()` and compare with `arePortfolioIdentitiesEqual()`.
- Do not add broker-detail, holdings, or order routes that skip `resolveTradingProviderContext()`, `resolveTradingProviderSelectedAccount()`, `checkSessionOrInternalAuth()`, and workspace access checks.
- Do not record submitted orders through a generic route or direct `orderHistoryTable` insert. Order writes must flow through `/api/providers/trading/order` and `recordOrderHistory()` so submission source, listing identity, workspace, and linked log validation stay consistent.
- Do not reintroduce direct workflow execution, `workflow_executor`, or the old `createStreamingResponse()` path. Use `enqueuePendingExecution()`, `runQueuedWorkflowExecution()`, `WorkflowExecutionEvent`, and the `/executions/[executionId]/stream` route.
- Do not make server-side workflow/tool execution depend on browser sessions. Use internal JWTs with `InternalWorkflowExecutionContext`, `checkSessionOrInternalAuth()`, and `authorizeCredentialUse()` for server-to-server routes.
- Do not add Copilot tools outside `COPILOT_TOOL_IDS`, `ToolArgSchemas`, `ToolResultSchemas`, `ToolRegistry`, and `apps/tradinggoose/stores/copilot/tool-registry.ts`. New read-only tools should follow the `read_*`/`get_*` naming split used here.
- Do not restore client-side wrappers for server-managed Copilot tools. Add display metadata in `SERVER_TOOL_METADATA` and route execution through `apps/tradinggoose/lib/copilot/tools/server/router.ts`.
- Do not invent parallel sub-block ids for the same execution parameter. Use `canonicalParamId`, `buildConfiguredSubBlockParams()`, `resolveDisplayedSubBlockValue()`, and `buildSubBlockRows()`.
- Do not mark secret, credential, broker-account, or user-picked values as `user-or-llm` to bypass validation. Keep them `user-only` or hidden, and let `tools/params.ts` keep LLM schemas focused on LLM-owned parameters.
- Do not bypass `selectedTriggerId` or revive fallback trigger resolution based on block type, hidden defaults, or webhook config. Extend `apps/tradinggoose/triggers/resolution.ts` and trigger config aggregation helpers instead.
