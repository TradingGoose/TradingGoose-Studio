### Key Changes
- Platform-managed runtime configuration is now modeled as service-specific rows instead of being mixed into global system settings. `packages/db/schema/system.ts` adds `systemServiceValue` and the `system_service_value_kind` enum, `apps/tradinggoose/lib/system-services/catalog.ts` defines the canonical OpenAI, Anthropic, Azure OpenAI, search, email, browser automation, Copilot API, and market API service fields, and `apps/tradinggoose/lib/system-services/service.ts` handles encrypted credential persistence plus typed setting resolution. The admin entrypoint lives in `apps/tradinggoose/lib/admin/system-services.ts`, `apps/tradinggoose/app/api/admin/services/route.ts`, and `apps/tradinggoose/app/admin/services/services-admin.tsx`.
- Global system settings were intentionally narrowed to platform flags and identity defaults. `apps/tradinggoose/lib/system-settings/service.ts` now owns registration mode, billing enablement, trigger-dev enablement, promotion-code allowance, and email-domain/from-address defaults only. Stripe runtime secrets were pulled out of that contract and now live in `apps/tradinggoose/lib/system-services/stripe-runtime.ts`, which reads deployment env vars through `apps/tradinggoose/lib/env.ts` instead of DB-backed settings.
- Better Auth system OAuth hydration is now request-scoped and provider-aware. `apps/tradinggoose/app/api/auth/[...all]/route.ts` detects only the callback/link/sign-in paths that need system-managed OAuth credentials, `apps/tradinggoose/lib/oauth/system-managed-config.ts` resolves those credentials from `apps/tradinggoose/lib/system-integrations/resolver.ts` and stores them in `AsyncLocalStorage`, and `apps/tradinggoose/lib/auth.ts` consumes them through lazily resolved provider getters instead of global mutation.
- Copilot chat persistence moved onto first-class review-session history tables and a single chat route. `packages/db/schema/copilot.ts` now defines `copilotReviewSessions`, `copilotReviewTurns`, and `copilotReviewItems`; `apps/tradinggoose/app/api/copilot/chat/route.ts` serializes streamed assistant text, tool calls, reasoning blocks, file/context attachments, and turn status into those tables; and `apps/tradinggoose/stores/copilot/store.ts` coordinates save/resume behavior through the split helpers in `apps/tradinggoose/stores/copilot/{store-messages,store-state,streaming,store-provenance,tool-registry,workspace-selection}.ts`.
- Copilot tool contracts are now generated from one canonical registry instead of ad hoc route-local shapes. `apps/tradinggoose/lib/copilot/registry.ts` owns `ToolIds` and `ToolArgSchemas`, `apps/tradinggoose/lib/copilot/tool-prompt-metadata.ts` owns user-facing tool semantics, `apps/tradinggoose/lib/copilot/runtime-tool-manifest.ts` converts those Zod contracts into JSON Schema, and `apps/tradinggoose/lib/copilot/runtime-tool-manifest-enrichment.ts` adds semantic validators for workflow Mermaid documents, entity documents, monitor documents, and embedded workflow sub-documents via `apps/tradinggoose/lib/copilot/workflow-subblock-semantic-contracts.ts`.
- Copilot billing and usage reservation logic was consolidated behind one API surface. `apps/tradinggoose/app/api/copilot/usage/route.ts` now handles reserve, adjust, commit, and release flows for both context and completion billing, while `apps/tradinggoose/lib/copilot/usage-reservations.ts` provides scope-aware reservation records and Redis locking keyed by user or organization billing scope. Runtime billing bypass still flows through `apps/tradinggoose/lib/billing/settings.ts` so disabled billing does not block Copilot usage.
- Workflow execution is now queue-first and billing-scope aware. `apps/tradinggoose/lib/execution/pending-execution.ts` owns enqueue, claim, retry, completion, and backlog limits; `apps/tradinggoose/background/pending-execution-drain.ts` dispatches pending workflow, webhook, schedule, indicator-monitor, and document jobs; `apps/tradinggoose/app/api/workflows/[id]/queue/route.ts` is the canonical queued workflow entrypoint; `apps/tradinggoose/lib/workflows/execution-runner.ts` centralizes workflow loading, environment decryption, usage checks, and executor startup; `apps/tradinggoose/lib/execution/execution-concurrency-limit.ts` enforces per-scope concurrency leases; and `apps/tradinggoose/services/queue/ExecutionLimiter.ts` is the renamed rate-limit owner for billed execution traffic.

### Design Decisions
- Provider/runtime configuration and platform flags are intentionally separate contracts now. Use `packages/db/schema/system.ts` plus `apps/tradinggoose/lib/system-services/{catalog,service}.ts` for provider-specific settings or secrets, and use `apps/tradinggoose/lib/system-settings/service.ts` only for cross-platform flags such as registration or billing enablement.
- Stripe is explicitly deployment-owned again. The replacement `apps/tradinggoose/lib/system-services/stripe-runtime.ts` reads `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` from env, while `apps/tradinggoose/lib/billing/settings.ts` only decides whether billing is enabled at runtime. Future code should not move Stripe secrets back into DB-backed system settings.
- Copilot runtime tool exposure is generated from one registry. Extend `ToolIds`, `ToolArgSchemas`, and `TOOL_PROMPT_METADATA`, then let `getCopilotRuntimeToolManifest()` derive JSON Schema and semantic validators. The branch deliberately removed parallel hand-maintained tool lists and route-local schema islands.
- Workflow document mutation now assumes a canonical Mermaid document format instead of permissive ad hoc shapes. `tg-mermaid-v1`, `TG_WORKFLOW` / `TG_BLOCK` / `TG_EDGE` metadata prefixes, canonical block contracts, and sub-block validation are the expected integration surface for both Copilot edits and live-state application.
- Async execution now flows through a backlog that is keyed by billing scope instead of having separate direct async code paths per trigger type. `enqueuePendingExecution()` plus the drain loop are the expected coordination layer, and concurrency/rate-limit checks are applied around that queue rather than at scattered call sites.
- Better Auth billing plans must be build-safe. `apps/tradinggoose/lib/billing/plans.ts` returns an empty static config during `phase-production-build` and only resolves DB-backed plans at runtime, which keeps auth setup from failing during Next production compilation.
